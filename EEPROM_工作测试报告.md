## 1. 测试目的

本次 EEPROM 功能测试的目标，是验证控制板在断电、重新上电、重新连接上位机等场景下，能够正确保存并恢复关键运行参数。

EEPROM 主要用于保存系统配置和运行状态，例如 LED 状态、PWM 参数、系统使能状态、风机使能状态、继电器命令状态以及相关保护状态信息。这样可以避免设备每次重新上电后丢失用户设置。

## 2. EEPROM 芯片与接口说明

当前工程使用外部 I2C EEPROM 作为非易失性存储器。MCU 通过 I2C 总线访问 EEPROM，实现参数的读写和上电恢复。

设计中将 EEPROM 作为配置数据的持久化存储区域，业务模块不直接操作 EEPROM 芯片，而是通过统一的配置管理模块完成保存和读取。这样可以减少不同业务模块重复写 EEPROM 的情况，也方便后续维护。

## 3. EEPROM 读写设计思路

EEPROM 读写功能主要分为三个层次：

1. 底层 EEPROM 驱动  
   负责完成 EEPROM 的在线检测、单字节读写、多字节读写、分页写入以及写入完成等待。

2. 配置存储管理  
   负责组织需要保存的系统参数，并在保存时增加版本号、序号、校验值和提交标志。

3. 业务功能调用  
   LED、PWM、系统使能、继电器等业务模块只更新配置数据，由配置管理模块统一决定何时写入 EEPROM。

这样设计的好处是 EEPROM 的读写细节不会分散在各个业务文件中，后续如果更换 EEPROM 芯片或调整存储格式，只需要集中修改驱动层和配置管理层。

## 4. 存储方案说明

### 4.1 多槽位存储【优化点】

为了提高 EEPROM 数据可靠性，当前采用多槽位轮换保存方案。

每次保存配置时，不是一直写同一个地址，而是轮流写入多个存储槽位。系统上电时会扫描所有槽位，选择校验正确且序号最新的一份配置作为当前有效配置。

这种方式可以带来两个好处：

- 减少固定地址反复擦写，降低 EEPROM 单一区域磨损。
- 如果某次写入过程中突然断电，旧的有效配置仍然可以被找到，不会导致全部配置丢失。

### 4.2 数据完整性设计

每份配置数据中包含以下信息：

| 字段 | 作用 |
|---|---|
| 固定标识 | 判断该区域是否为有效配置数据 |
| 版本号 | 用于区分不同版本的配置结构 |
| 数据长度 | 判断数据长度是否符合当前程序要求 |
| 保存序号 | 判断哪一份配置是最新保存的 |
| 参数数据 | 保存实际业务参数 |
| CRC 校验 | 判断 EEPROM 中的数据是否损坏 |
| 提交标志 | 判断本次写入是否完整完成 |

上电加载时，程序会同时检查固定标识、版本号、数据长度、CRC 校验和提交标志。只有全部检查通过的数据，才会被认为是有效配置。

### 4.3 掉电保护思路

EEPROM 写入不是瞬间完成的，如果写入过程中掉电，可能只写入了一部分数据。

因此当前设计中不会只依赖单一配置区，而是**保留多份历史配置**。保存新配置时，只有当整份数据写入完成并写入提交标志后，该槽位才被认为有效。

如果写入中途掉电，新槽位可能校验失败或提交标志无效。下一次上电时，程序会自动跳过这份损坏数据，继续使用上一份有效配置。

## 5. 参数保存策略【优化点】

EEPROM 的写入寿命有限，不能每次参数变化都立即写入。

当前设计采用“延时保存”的策略：

1. 当业务参数发生变化时，先更新 RAM 中的配置数据。
2. 标记当前配置为“待保存”。
3. 等待一段时间，如果这段时间内参数继续变化，只刷新 RAM 数据，不重复写 EEPROM。
4. 到达保存时间后，一次性将最终配置写入 EEPROM。

这样可以避免用户连续调节 PWM 或频繁开关 LED 时导致 EEPROM 被高频写入。

## 6. EEPROM 功能测试内容

### 6.1 在线检测测试

测试方法：

- 上电初始化 EEPROM 驱动。
- 调用 EEPROM 在线检测流程。
- 判断 EEPROM 是否能够正常响应 I2C 地址。

预期结果：

- EEPROM 在线检测成功。
- 系统允许进入配置读取流程。

### 6.2 写入与读取一致性测试

测试方法：

- 修改一个需要保存的参数，例如 LED 状态或 PWM 值。
- 触发配置保存。
- 重新读取 EEPROM 中的数据。
- 判断读取结果是否与写入前的配置一致。

预期结果：

- 写入后可以正确读回。
- 读回数据与保存前的数据一致。
- CRC 校验通过。

### 6.3 上电恢复测试

测试方法：

- 设置 LED 状态、PWM 参数等配置。
- 等待配置保存完成。
- 断电后重新上电。
- 通过上位机读取当前状态。

预期结果：

- 上电后系统能够自动加载 EEPROM 中最新有效配置。
- LED、PWM 等参数恢复为断电前保存的状态。
- 上位机重新打开后读取到的是板端实际保存状态，而不是上位机默认状态。

### 6.4 异常数据保护测试

测试方法：

- 模拟 EEPROM 中某一份配置数据无效，例如 CRC 不匹配、版本不匹配或提交标志不正确。
- 重新上电加载配置。

预期结果：

- 程序不会使用无效配置。
- 如果还有其他有效槽位，则自动加载最新有效槽位。
- 如果没有任何有效配置，则使用默认参数初始化系统。



## 9. 调试过程中遇到的问题与解决过程



### 9.1 保存过于频繁的问题

在 PWM、LED 等参数调试时，参数可能会被连续修改。如果每次修改都立即写 EEPROM，会导致 EEPROM 写入次数过多。

解决方式：

- 采用“参数变化后延时保存”的策略。
- 在延时时间内如果参数继续变化，只更新 RAM 中的最终值。
- 延时结束后再统一写入 EEPROM。

这样既能保证最终参数被保存，又能减少 EEPROM 的无效写入。

### 9.2 写入过程中掉电可能导致数据不完整

EEPROM 写入需要内部写周期，如果写入过程中突然断电，当前正在写入的数据可能只写入了一部分。

解决方式：

- 使用多个配置槽位轮换保存。
- 每份配置增加 CRC 校验。
- 每份配置增加提交标志。
- 上电加载时只使用校验正确且提交完整的数据。

这样即使某一次保存中途掉电，系统也可以回退到上一份有效配置。

### 9.3 今日开关量调试补充

今天对 K0 开关量检测逻辑进行了联调。根据原理图确认，开关量通过 74HC4051 进行通道复用，MCU 需要先设置选择线，再读取公共输入端电平。

调试时重点确认了两点：

- 端子短接时，对应开关量应识别为闭合状态。
- 端子断开时，对应开关量应识别为断开状态。

同时补充了软件消抖思路，业务逻辑不直接使用瞬时 IO 电平，而是使用连续多次一致后的稳定状态，避免导线接触抖动导致状态误变化。后续高压开关保护也会基于这个稳定开关量状态进行判断。

## 10. 测试结论

当前 EEPROM 方案采用底层驱动、配置管理和业务调用分层设计，能够实现关键参数的持久化保存。

通过多槽位轮换、CRC 校验、提交标志和延时保存机制，可以提高掉电场景下的数据可靠性，同时减少 EEPROM 的无效写入次数。

该设计适合当前控制板保存 LED 状态、PWM 参数、系统使能状态和继电器命令状态等配置，为后续功能扩展提供了统一的参数持久化基础。
